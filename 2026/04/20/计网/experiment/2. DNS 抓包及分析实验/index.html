Plaintext

  • 原因： 这意味着该结果来自缓存。提供这个答案的服务器（即你的本地解析器或运营商 DNS）并不是 mit.edu 域名的官方管理员。它只是“听说”过这个答案并把它存了下来，直接告诉你。

3. 别名的“套娃” (Canonical Name / CNAME)

这是这段结果中最有趣的部分。CNAME (Canonical Name) 相当于给域名取了“外号”：

  1. 你查 www.mit.edu。

  2. DNS 告诉你：“它其实叫 www.mit.edu.edgekey.net。”

  3. DNS 又告诉你：“而那个家伙其实叫 e9566.dscb.akamaiedge.net。”

为什么这么麻烦？ 因为 MIT 使用了 Akamai（全球最大的 CDN 服务商之一）。通过这种别名跳转，Akamai 就可以根据你（客户端）所在的地理位置，动态地把你引导到离你最近、速度最快的缓存服务器上。

  • IPv6 地址： 后面那些带冒号的长字符串（如 2600:1417...）。这说明 MIT 的网站已经支持了 IPv6 访问。

www.zju.edu.cn 后面跟了一串很长的“套娃”–别名 (Canonical Name)：

  1. www.zju.edu.cn -> ...w.cdngslb.com

  2. ...w.cdngslb.com -> ...queniusa.com

  3. ...queniusa.com -> ...v3.so666gslb.com

这说明浙大官网使用了全局服务器负载均衡 (GSLB) 和 CDN 加速。

  • gslb（Global Server Load Balancing）：这是为了确保全国各地的同学访问官网时，系统能根据你的位置，把你分配到响应最快的节点。

  • so666gslb.com 可能是学校合作的特定加速服务商的域名。

1. IPv4 (Internet Protocol version 4)

IPv4 是我们最熟悉的格式，采用点分十进制表示法。

  • 结构： 总共 32 位 (bit)，分为 4 个字节 (Octets)。

  • 格式规定：

    • 由 4 个 0 到 255 之间的数字组成。

    • 中间用三个点 (.) 隔开。

  • 范围： 0.0.0.0 到 255.255.255.255。

  • 例子： 你的结果 111.32.202.196。

    • 每个数字其实代表 8 位二进制。比如 111 对应二进制 01101111。

2. IPv6 (Internet Protocol version 6)

由于 IPv4 地址快用光了，IPv6 采用了更庞大的冒号分十六进制表示法。

  • 结构： 总共 128 位 (bit)，分为 8 组。

  • 格式规定：

    • 每组由 4 个十六进制数（0-9, a-f）组成。

    • 中间用冒号 (:) 隔开。

  • 范围： 非常巨大，理论上可以给地球上的每一粒沙子分配一个 IP。

  • 例子： 2409:8c02:21c:a3:3::19。

查询结果里的 :: 就是 IPv6 的缩写技巧，这在实验报告里经常被考到：

  1. 省略前导零： 每组开头的 0 可以省略。

    • 例如：0003 可以写成 3。这就是为什么你查到的地址里有 :a3:3: 这样的部分。

  2. 双冒号 (::) 压缩： 如果连续几组都是 0000，可以用一个 :: 代替。

    • 硬性规定： 在一个地址中，:: 只能出现一次（否则电脑不知道中间到底缺了几组零）。

    • 还原： 你的地址 ...:3::19。因为总共有 8 组，现在能看到的只有 6 组，说明 :: 代表了中间有两个连续的 0000:0000。

2. Run nslookup to determine the authoritative DNS servers for a university in Europe.

  • 找出一家欧洲大学的权威 DNS 服务器（Authoritative DNS Servers）

使用交互模式（更清晰）

  1. 在终端输入 nslookup 进入交互模式。

  2. 输入 set type=NS（告诉工具你要查的是名称服务器，而不是 IP）。

  3. 输入大学域名，例如 ox.ac.uk。

结果解析

  • 权威服务器名单： 牛津大学配置了 6 台服务器来负责 DNS 解析：

    • dns0.ox.ac.uk 到 dns2.ox.ac.uk

    • auth4.dns.ox.ac.uk 到 auth6.dns.ox.ac.uk

  • 命名规范： 注意到它们分成了 dns 和 auth.dns 两组，这通常意味着学校在内部管理和外部公开查询上做了某种架构上的隔离（比如负载均衡或主从备份）。

为什么这里还是显示 “Non-authoritative answer”？

这是一个很多同学会混淆的点：为什么我查的是 NS 记录（权威服务器），结果还是“非权威回答”？

  • 原因： 因为你刚才问的是 127.0.0.53（你的本地缓存管家）。

  • 逻辑： 本地管家说：“哦，你想知道牛津的权威服务器是谁啊？我之前刚好帮别人查过，我直接从我的缓存里把这几个名字告诉你吧。”

  • 结论： 只要是从缓存里拿出来的，哪怕内容是“谁是权威”，这个动作本身依然是非权威的。

权威 DNS 服务器（Authoritative Name Server）通常不兼职做递归解析器（Recursive Resolver）。

  • 递归查询限制（Recursion Restriction）： 大多数机构（如大学、公司）的权威 DNS 服务器为了安全和节省资源，只为自家校园网内部用户提供递归查询（帮你查外部网站如 Yahoo）。对于来自校外（比如你所在的网络）的请求，它只负责回答关于 ox.ac.uk 的问题。如果你问它 yahoo.com，它会觉得“这不归我管，我也没义务帮你去问别人”，所以直接返回 REFUSED。

  • 防火墙拦截（Firewall/ACL）： timed out 提示说明你的查询包可能直接被对方的防火墙拦截了。

4. Locate the DNS query and response messages. Are then sent over UDP or TCP?

  • 定位 DNS 查询和响应消息。它们是通过 UDP 还是 TCP 发送的？

可见是通过 UDP 发送的。

DNS 协议设计之初就优先考虑了速度和开销。

  • 低延迟： UDP 是无连接的，不需要像 TCP 那样进行“三次握手”。对于像“查个地址”这样的小型往返任务，UDP 快得多。

  • 报文小： 传统的 DNS 查询和响应通常小于 512 字节，UDP 足以容纳。

  • 服务器负担： DNS 服务器每秒要处理海量请求，如果每个请求都要维持一个 TCP 连接，服务器的资源会迅速枯竭。

11. What is the destination port for the DNS query message? What is the source port of DNS response message?

  • DNS 查询消息的目标端口是多少？DNS 响应消息的源端口是多少？

DNS 查询消息 (Query, 以 No. 4006 为例)

  • 目的端口 (Destination Port): 53

  • 分析： 所有的 DNS 查询请求都会发送到服务器的 53 端口，这是全球统一的标准。你的笔记本从随机高位端口（如 51727 等）发出请求，目标指向服务器的 53 端口。

DNS 响应消息 (Response, 以 No. 4007 为例)

  • 源端口 (Source Port): 53

  • 分析： DNS 服务器在处理完你的请求后，会从它的 53 端口发出回包。对于你的电脑来说，它看到的这个包的“出发地”就是 53 端口。
